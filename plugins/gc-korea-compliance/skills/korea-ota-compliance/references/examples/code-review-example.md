# Worked Example: Code Review — BookingController PII Logging

<!-- Last reviewed: 2026-04-09 -->

This example walks through a complete compliance code review cycle for a Spring Boot controller
that inadvertently logs personal information. It shows how to classify the scope, generate
findings at the right severity, and summarise results.

---

## Input

**Artifact type:** Code review
**Surface:** `services/domain-api` — Spring Boot BookingController

```java
// BookingController.java
@RestController
@RequestMapping("/api/v1/bookings")
@RequiredArgsConstructor
public class BookingController {

    private static final Logger log = LoggerFactory.getLogger(BookingController.class);
    private final BookingService bookingService;

    @PostMapping
    public ResponseEntity<BookingResponse> createBooking(
            @RequestBody @Valid BookingRequest request,
            @AuthenticationPrincipal TravelerPrincipal principal) {

        log.info("Creating booking for traveler: {}", request);   // line 16

        BookingResponse response = bookingService.createBooking(
                principal.getTravelerId(), request);

        return ResponseEntity.status(HttpStatus.CREATED).body(response);
    }
}
```

```java
// BookingRequest.java
public record BookingRequest(
        String travelerName,     // 여행자 이름
        String travelerPhone,    // 여행자 전화번호
        String travelerEmail,    // 여행자 이메일
        String passportNumber,   // 여권번호
        String nationality,      // 국적
        LocalDate birthDate,     // 생년월일
        Long productId,
        Long optionId,
        LocalDate tourDate,
        int quantity
) {}
```

**PR description:** "Add booking creation endpoint. Accepts traveler info and product/option
selection, delegates to BookingService for confirmation logic."

---

## Step 1 — Classify

Before applying any rules, determine what kind of work this is and which rule sets apply.

| Dimension | Assessment |
|---|---|
| **Business role** | Intermediary — the OTA facilitates bookings between travelers and partner suppliers. It is not the merchant of record. |
| **Data classification** | High-risk PII is present: `travelerName`, `travelerPhone`, `travelerEmail`, `passportNumber`, `nationality`, `birthDate`. `passportNumber` is 고유식별정보 under PIPA. |
| **Payment role** | Not visible in this controller; payment is handled by the shared Payment Platform. No EFTA / card-handling rules apply here. |
| **Scope gates active** | PIPA (personal data handling), ISMS-P (logging, access control). E-Commerce Act and Tourism Act disclosure rules are not triggered by this controller alone. |
| **Location data** | Not present. LIA rules do not apply. |
| **Overseas transfer** | Not evident. Transfer-register rules not triggered by this snippet. |

Rule sets loaded for this review: **PIPA**, **ISMS-P**.

---

## Step 2 — Findings

### Finding 1

```text
Severity : BLOCK
Rule     : PIPA-001 — Raw PII in application logs
Location : BookingController.java, line 16
```

**What the code does:** `log.info("Creating booking for traveler: {}", request)` calls
`BookingRequest.toString()` (auto-generated by the Java `record` compiler). This serialises
all fields — including `travelerName`, `travelerPhone`, `travelerEmail`, `passportNumber`,
`nationality`, and `birthDate` — into the log line in plain text.

**왜 문제인가 (Why this matters):**
개인정보보호법 제29조는 개인정보처리자가 개인정보의 안전성 확보에 필요한 기술적·관리적 보호조치를 취하도록 요구합니다.
애플리케이션 로그에 성명, 전화번호, 이메일, 여권번호, 생년월일이 평문으로 기록되면,
로그 파일에 접근할 수 있는 모든 인원(운영자, 인프라 팀, 로그 수집 시스템)이 이를 열람할 수 있습니다.
위반 시 개인정보보호위원회로부터 과징금(최대 전년도 매출액의 3%), 시정명령, 공표 조치가 부과될 수 있습니다.
또한 ISMS-P 인증 심사에서 중결함으로 분류되어 인증 취소 또는 보완 조치가 요구될 수 있습니다.

**Fix options:**
1. Use a `MaskUtils` helper to log only safe identifiers:
   ```java
   log.info("Creating booking — travelerId={}, productId={}, optionId={}",
            principal.getTravelerId(), request.productId(), request.optionId());
   ```
2. Implement a structured logging filter that redacts fields matching known PII keys before
   writing to the appender.
3. Override `toString()` on `BookingRequest` (or add a custom `@JsonIgnore`-equivalent method)
   that returns only non-sensitive fields — but prefer option 1 or 2 as a defence-in-depth layer.

---

### Finding 2

```text
Severity : BLOCK
Rule     : PIPA-002 — Sensitive identifier (고유식별정보) without verified encryption
Location : BookingRequest.java — passportNumber field
```

**What the code does:** `passportNumber` is collected as a plain `String` with no annotation,
validation pattern, or comment indicating it is encrypted before persistence.

**왜 문제인가 (Why this matters):**
여권번호는 개인정보보호법 시행령 제19조에서 정의하는 고유식별정보에 해당합니다.
동법 제24조는 고유식별정보를 처리할 때 암호화 등 안전성 확보 조치를 의무화합니다.
DTO가 평문으로 수신된 뒤 암호화 없이 도메인 레이어나 데이터베이스에 전달될 경우,
인증 위반에 해당하며 위반 1건당 최대 5,000만 원의 과태료 대상이 됩니다.

**Fix options:**
1. Confirm that `BookingService.createBooking()` calls an encryption utility before persisting
   `passportNumber`. If so, add a code comment and test asserting the encrypted value is stored.
2. If encryption does not happen in `BookingService`, add it there before calling the repository.
3. Consider whether `passportNumber` is truly required at booking creation or only at voucher
   generation (deferred collection reduces risk surface).

---

### Finding 3

```text
Severity : WARN
Rule     : PIPA-004 — Collection without verified consent basis for optional fields
Location : BookingRequest.java — nationality, birthDate fields
```

**What the code does:** `nationality` and `birthDate` are collected as part of every booking
request. It is not clear from this code whether these fields are required for all tour types
or whether they are optional, and whether a separate consent checkbox covers their collection.

**왜 문제인가 (Why this matters):**
개인정보보호법 제15조는 개인정보 수집 시 수집 목적, 항목, 보유 기간을 명시하고
정보주체로부터 동의를 받도록 규정합니다.
국적 및 생년월일이 일부 투어(예: 특정 국가 입장 제한, 연령 제한 투어)에만 필요한 경우,
이를 모든 예약 요청에서 수집하는 것은 최소 수집 원칙(비례성 원칙)에 위배될 수 있습니다.

**Fix:** Confirm with the product team:
- Is `nationality` required for all day tours, or only for tours with nationality-based entry
  restrictions?
- Is `birthDate` required for all tours, or only for age-restricted products?
- Is separate consent obtained for each optional field on the checkout UI?

If either field is optional for some tours, make it nullable and validate presence only when the
product configuration requires it.

---

### Finding 4

```text
Severity : INFO
Rule     : ISMS-003 — Audit trail for booking creation not visible in this layer
Location : BookingController.java
```

**What the code does:** The controller delegates to `BookingService` but does not inject a
correlation ID or structured audit event into the request context before doing so.

**Suggestion:** Add a correlation ID (e.g., `MDC.put("correlationId", UUID.randomUUID())`) at
the controller entry point so that all downstream log statements for this booking creation are
traceable to a single request. This also aids ISMS-P audit trail requirements for high-risk
operations.

This is informational — it does not block approval, but should be addressed before launch.

---

## Step 3 — Uncertain findings

None. All findings above are based on directly observable code behaviour.

If the reviewer cannot confirm whether `BookingService` encrypts `passportNumber` (Finding 2),
that finding remains **BLOCK** until encryption is verified in the service layer.

---

## Step 4 — Summary

| Severity | Count | Finding IDs |
|---|---|---|
| BLOCK | 2 | PIPA-001, PIPA-002 |
| REVIEW | 0 | — |
| WARN | 1 | PIPA-004 |
| INFO | 1 | ISMS-003 |

**Verdict:** This PR must not be merged until both BLOCK findings are resolved.

**Blocker resolution checklist:**
- [ ] Remove raw `request` object from log statement (PIPA-001)
- [ ] Confirm or add encryption for `passportNumber` in `BookingService` (PIPA-002)

**Escalations required:** 0

**Watchlist for follow-up (not blocking this PR):**
- Confirm consent basis and optionality for `nationality` and `birthDate` (PIPA-004) before
  checkout UI is finalised.
- Add correlation ID to request context (ISMS-003) before launch.
